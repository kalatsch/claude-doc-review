# claude-doc-review — дизайн плагина

**Статус:** черновик на ревью · **Дата:** 2026-06-19

## Цель

Плагин превращает **технический документ, который Claude написал для себя**
(план/спека/анализ — плотный, как `PLAN.md`), в **человеческую версию для
ревью**: понятную прозу + словарь терминов + сворачиваемые технические детали
(«под кат») + слой комментирования (выделил текст → тред → Claude отвечает →
resolve, с синком через локальный сервер). Документ любой, не только план.

Ключевая идея — два уровня одного документа: Claude ведёт свой технический,
а человек смотрит очеловеченную версию и комментирует её; комментарии
синкаются обратно к Claude.

«Done» = в любом проекте можно выполнить `/doc-review путь/к/документу.md`,
получить понятную страницу (проза + словарь + свёрнутая детализация) с рабочим
слоем комментирования и локальным сервером, и вести в комментариях диалог с
Claude, который пишется обратно в файл.

## Не-цели

- Не рендерим plan-специфику (file-tree, диаграммы, wireframe-канвас,
  Open-Questions-форму) — нужен универсальный документ.
- Не хостим ничего в облаке, нет аккаунтов/шаринга между машинами — всё локально
  в рамках машины (как у BuilderIO local-files mode, но проще).
- Не редактируем сам документ из UI в v1 (комментарии — да, тело документа — нет).
- Не поддерживаем HTML-вход — только markdown.
- **Нет развилки режимов** на старте — команда просто готовит файл и поднимает
  сервер. (Раньше закладывался выбор «живой диалог / разовая генерация» — убран:
  при простом «добавить готовый файл» он лишний.)

## Форма и упаковка

**Claude Code плагин** (как авторский `claude-activity`), репозиторий
`~/Desktop/GitLab/claude-doc-review-plugin/`. Имя плагина и команда —
`claude-doc-review` / `/doc-review` (зафиксировано).

Главная фича — **slash-команда в основном цикле**, а не спавн-сабагент:
очеловечивание документа и живой диалог в комментариях ведёт основной цикл
Claude (сабагент отработал бы и завершился). Поэтому это команда + ассеты-рендерер,
не агент.

## Архитектура

Три части, все — ассеты плагина, копируются в проект при запуске:

### 1. `review.html` — рендерер (generic shell)

Один файл, **источник правды для вида**. Порт движка из `plan.html`, очищенный
от plan-специфики. Принцип BuilderIO: «критикуют вид — чини рендерер, не правь
сохранённый артефакт». Поэтому `review.html` универсален и **не содержит самого
документа** — он на клиенте подтягивает `human.md` и рендерит его.

Состав:
- **Markdown → HTML на клиенте** мини-парсером (вендорим `marked.min.js` как
  ассет плагина, без CDN — работает офлайн). Рендерит `human.md`. Редактируешь
  `human.md` → перезагрузка страницы перерисовывает.
- **Слой комментирования** (ядро, из `plan.html`): выделение текста → плавающая
  кнопка → тред; якоря по текстовым офсетам с фолбэком на `quote` + контекст
  (`contextBefore`/`contextAfter`, идея из BuilderIO `get-plan-feedback`) для
  переякоривания при правках; «осиротевшие» треды помечаются, не теряются.
- **Треды**: вложенные ответы, авторы user/claude, resolve/reopen, удаление —
  как сейчас.
- **Two-axis state** (из BuilderIO): у сообщения Claude есть пометка `consumed`
  (Claude учёл коммент) отдельно от `resolved` (тема закрыта) — чтобы Claude
  видел, какие комменты ещё не обработаны.

Очеловечивание (новое — суть плагина):
- **Словарь терминов**: термин в тексте получает пунктирное подчёркивание +
  тултип с определением; отдельная секция «Словарь» внизу. Источник — секция
  глоссария в очеловеченном markdown (см. шаг команды).
- **«Под кат» (сворачиваемые детали)**: очень технические/детальные куски
  прячутся в `<details>`-блок «показать детали», чтобы человеческая версия
  оставалась читаемой. Marked пропускает сырой HTML, поэтому `<details>` в
  markdown работает из коробки.

Улучшения «красивее/удобнее» (общие, без plan-специфики):
1. **Полировка вида** — типографика, плотность, тени/радиусы, переключатель
   светлая/тёмная тема, печать в PDF.
2. **Навигация** — оглавление/мини-карта по заголовкам документа, индикатор
   «N открытых комментов», кнопка «следующий открытый коммент», прогресс
   «решено N из M».
3. **UX комментинга** — плавающая кнопка у выделения (есть), markdown в тексте
   комментариев, аккуратные треды, бейдж «не обработано Claude».
4. **Шапка статуса** — название документа + статус (черновик / на ревью /
   готово) + счётчики открытых/решённых.

### 2. `serve.js` — локальный сервер

Порт из `screens-admin/.claude/plan/serve.js` почти без изменений:
- статика из своей папки;
- `GET/POST /api/comments` ↔ `comments.json` (с `X-Version` по mtime для
  polling-синка);
- дополнительно отдаёт `human.md` (read-only) — чтобы `review.html` его
  подтянул и отрендерил.

Fallback-режим (открыть `review.html` файлом, без сервера) с экспортом/импортом
JSON — сохраняем, как в оригинале.

### 3. Команда `/doc-review` — оркестрация

Инструкция в основном цикле. Без развилки режимов. Шаги:
1. Принять аргумент — путь к техническому `.md` (документ Claude «для себя»).
   Если не задан — попросить.
2. **Очеловечить документ.** Claude читает технический исходник и пишет
   `human.md`: понятная проза вместо плотного жаргона; секция «Словарь» с
   определениями терминов; очень технические/детальные куски заворачиваются в
   `<details>`-блоки «под кат». Исходник не меняется — это отдельный
   человеческий слой.
3. Вычислить `<slug>` из имени файла. Создать `.claude/info/<slug>/` в проекте.
4. Скопировать туда `review.html`, `serve.js`, `marked.min.js`; положить
   `human.md` (очеловеченная версия — её и рендерит страница), `source.md`
   (копия технического исходника, для справки) и пустой `comments.json`
   (`{"version":1,"threads":[]}`), если его ещё нет.
5. Запустить `node serve.js`, отдать пользователю URL.
6. Дальше Claude в этой же сессии по ходу диалога читает `comments.json`,
   отвечает на необработанные комменты пользователя, помечает `consumed` / при
   необходимости `resolved`, пишет файл обратно. (Это естественное продолжение
   сессии, а не выбранный заранее «режим».)

**Расположение артефактов:** `.claude/info/<slug>/` в проекте, где запущена
`/doc-review`. Несколько документов = несколько подпапок, не конфликтуют.

## Поток данных

```
технический.md ─/doc-review─► Claude очеловечивает ─► .claude/info/<slug>/
                                                         ├─ human.md        (человеческая версия — рендерится)
                                                         ├─ source.md       (копия тех. исходника, справка)
                                                         ├─ review.html     (рендерер + слой комментов)
                                                         ├─ marked.min.js   (md→html на клиенте)
                                                         ├─ serve.js         (статика + comments API)
                                                         └─ comments.json    (треды; читает/пишет и Claude, и UI)

браузер ◄─ HTML+комменты ─ serve.js ◄─ poll/POST ─► comments.json ◄─ читает/пишет ─ Claude (осн. цикл)
```

## Что заимствуем у BuilderIO (github.com/BuilderIO/skills, /visual-plan)

Точечно, без их хостинг-инфраструктуры:
- **Робастность якорей**: `quote` + `contextBefore`/`contextAfter` для
  переякоривания и явная пометка «detached» вместо тихой потери треда.
- **Two-axis state**: `consumed` (учтено агентом) ≠ `resolved` (закрыто).
- **Quality-bar рендера документа**: чистая типографика, без маркетингового
  мусора (их `document-quality.md`).

НЕ берём: structured-blocks, canvas/prototype, question-form, MCP-коннектор,
аккаунты, экспорт в их формат.

## Риски и открытые вопросы

- **Markdown-парсер на клиенте**: офсеты якорей зависят от детерминированности
  рендера. `marked` детерминирован; фолбэк по quote+контексту страхует правки.
- **Очеловечивание — лишний шаг переписывания**: Claude тратит токены на
  `human.md`. Принимаем — это и есть ценность плагина. Исходник не трогаем.
- **Ответы на комменты** перечитываются по ходу сессии, не фоновым таймером.
  (Позже можно прикрутить `/loop`/ScheduleWakeup, но в v1 не нужно.)
- ~~Имя плагина/команды~~ → зафиксировано: `claude-doc-review` / `/doc-review`.
- ~~Подпапка `<slug>` внутри `.claude/info/`~~ → зафиксировано: подпапка под
  каждый документ.

## Проверка (e2e smoke)

В тестовом проекте: `/doc-review test-doc.md` → проверить, что `human.md` понятнее
исходника, есть «Словарь» и хотя бы один `<details>`-блок → открыть URL →
тултип термина работает → раскрыть «под кат» → выделить текст → оставить коммент
→ убедиться, что он попал в `comments.json` → Claude отвечает → ответ виден в
браузере после polling → resolve → перезагрузка сохраняет состояние. Плюс
fallback: открыть `review.html` файлом, проверить экспорт/импорт.
